# Repository Guidelines for Agentic AI

This document outlines the conventions, commands, and rules for AI coding agents operating within this repository. Adhere strictly to these guidelines to maintain consistency and stability.

## 1. Project Overview & Tech Stack
- **Framework:** React 19 + Vite 6
- **Language:** TypeScript (ES2022 target)
- **Styling:** Tailwind CSS v4, `clsx`, `tailwind-merge`
- **Icons:** `lucide-react`
- **Animation:** `motion/react`
- **Backend & Data:** Supabase (Auth & Database), better-sqlite3 (for local ingestion scripts)

## 2. Directory Structure
- `src/pages/` - Route-level page components (e.g., `CityPage.tsx`, `Home.tsx`).
- `src/components/` - Reusable UI components (e.g., `BusinessCard.tsx`).
- `src/contexts/` - React Context providers (e.g., `AuthContext.tsx`).
- `src/lib/` - Third-party integrations and singletons (e.g., `supabase.ts`, `auth.ts`).
- `src/lib/analytics.ts` - Lightweight browser analytics helpers (e.g., `trackEvent`).
- `src/lib/recommendations.ts` - Rule-based owner recommendation logic for claim and dashboard flows.
- `src/data.ts` - Seeded and static directory content.
- `src/business.ts` - Core business logic, type definitions, and helper utilities.
- `scripts/` - Node scripts (via `tsx`) for data fetching, generation, and DB synchronization.
- `dist/` - Production build output (git-ignored, generated by Vite).

## 3. Development Commands

### Building & Linting
Run these commands to ensure the codebase remains clean and compilable:
```bash
npm install       # Install dependencies
npm run dev       # Start local development server on port 3000
npm run build     # Create production build in dist/
npm run preview   # Serve production build locally
npm run clean     # Remove dist/ directory
npm run lint      # Run TypeScript type-checking without emitting files (tsc --noEmit)
```

### Testing (Vitest + React Testing Library)
There is no automated test suite by default, but if you install Vitest or are asked to run tests, use the following:
```bash
# Run all tests
npx vitest run

# CRITICAL: How to run a single test file
npx vitest run src/components/BusinessCard.test.tsx

# Run a specific test by name match
npx vitest run -t "should render properly"

# Run tests in watch mode for development
npx vitest watch
```

### Data & Database Synchronization
```bash
npm run import:serper         # Import data from Serper API
npm run merge:generated       # Merge generated business data
npm run enrich:serper         # Enrich data with Serper information
npm run export:gmaps          # Export Google Maps scraper input
npm run enrich:gmaps          # Enrich data using GMaps scraper results
npm run export:supabase-seed  # Export seed data for Supabase
npm run sync:supabase         # Sync local SQLite data with Supabase
```

## 4. Code Style & Conventions

### Formatting & Syntax
- Use **2 spaces** for indentation, especially in JSX/TSX files.
- **Semicolons** are mandatory at the end of statements.
- Use single quotes for standard strings, double quotes for JSX attributes (e.g., `className="text-red-500"`).

### Naming Conventions
- **Components & Pages:** `PascalCase` (e.g., `FeatureCard.tsx`).
- **Functions & Variables:** `camelCase` (e.g., `fetchBusinessData`, `isActive`).
- **Constants:** `UPPER_SNAKE_CASE` for global/magic values (e.g., `MAX_RETRY_COUNT`).
- **Types & Interfaces:** `PascalCase`. Do not use `I` or `T` prefixes (e.g., prefer `BusinessProfile` over `IBusinessProfile`).
- **Routes & CSS Classes:** `kebab-case` (e.g., `/contact-us`, `bg-blue-500`).

### Imports Organization
Always group your imports with spacing between groups to maintain readability:
1. Third-party packages (`react`, `react-router-dom`, `clsx`, `lucide-react`)
2. Contexts and library singletons (`@/src/lib/supabase`, `@/src/contexts/AuthContext`)
3. Internal React components (`@/src/components/UserMenu`)
4. Data and utility functions (`@/src/business`, `@/src/data`)
5. Styles and assets (`@/src/index.css`, images)

Always use the `@/` alias (which resolves to the project root) for absolute paths. Avoid long relative paths like `../../../`. Prefer named imports over default imports unless required by the library.

## 5. TypeScript & React Patterns

### Component Architecture
- **Functional Components:** Use function declarations for React components. Do NOT use `React.FC` or class components.
- **Props:** Destructure props directly in the function signature. Always define an `interface` for props.
- **Early Returns:** Handle loading, error, and empty states at the top of your component using early returns. Avoid deeply nesting conditional logic in the main return statement.
- **List Keys:** Use stable, unique IDs for the `key` prop when mapping arrays. Do NOT use array indices for keys unless the list is completely static.
- **Memoization:** Strategically use `useMemo` for heavy computations or filtering arrays, and `useCallback` for functions passed as props to child components.
- **Shared Decision Logic:** Put reusable recommendation or routing heuristics in `src/lib/` helpers instead of duplicating conditional CTA logic across multiple pages.

### Styling (Tailwind v4)
- Exclusively use **Tailwind CSS v4** utility classes for styling. Do not use inline `style={{}}` unless dynamically calculating values that Tailwind cannot handle.
- Manage dynamic classes with `clsx` combined with `tailwind-merge` to prevent specificity conflicts.
- Prefer semantic layout structures (e.g., `<main>`, `<section>`, `<article>`).

### Animation (Motion)
- Animate elements using `import { motion } from 'motion/react'`.
- Define animation configuration objects (variants) outside the component definition to keep JSX clean.
- Wrap conditionally rendered animated components in `<AnimatePresence mode="wait">`.

## 6. Error Handling & Defensive Programming

- **Null/Undefined Safety:** Assume data can be missing. Liberally use optional chaining (`?.`) and nullish coalescing (`??`). Example: `user?.profile?.name ?? 'Guest'`.
- **Async Operations:** Wrap all asynchronous data fetching (e.g., Supabase queries) in `try...catch` blocks.
- **User Feedback:** Log errors to `console.error` and ensure a user-friendly error message is displayed in the UI when things fail. Avoid blank screens on failure.
- **Security:** External links must use `<a target="_blank" rel="noopener noreferrer">` to prevent reverse tabnabbing attacks. Never hardcode API keys or sensitive endpoints; rely on `import.meta.env`.
- **Browser Globals:** When writing browser integrations such as `window.dataLayer`, initialize the value on `window` before mutating it. Example: `window.dataLayer = window.dataLayer || []; window.dataLayer.push(...)`.
- **Fallbacks:** Render fallback UI (`return null;` or a skeleton component) when required data is not available yet.

## 7. AI Agent Execution Rules

1. **Verify Your Work:** You MUST run `npm run lint` and `npm run build` locally within the environment after making changes to verify that no TypeScript or compilation errors have been introduced.
2. **Minimal Changes:** Only refactor files strictly necessary to complete the current user request. Maintain existing project conventions over rewriting them.
3. **No Test Proliferation:** Do not proactively write or generate test files unless the user explicitly requests tests to be created.
4. **Environment Configurations:** Do not modify `vite.config.ts` or `tsconfig.json` unless absolutely critical for resolving a user task. HMR is disabled intentionally. Keep it disabled.
5. **Placeholder Assets:** When implementing new features, use `lucide-react` for icons or simple CSS shapes for images rather than relying on external image URLs that may break.
